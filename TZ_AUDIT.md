# TZ.md — Аудит проекта Astro Bot и план доведения до MVP

# Общая оценка

Текущее состояние проекта: **45–55% готовности к MVP**.

Плюсы:
- Есть базовая архитектура.
- Используется aiogram 3.x.
- Есть PostgreSQL через SQLAlchemy.
- Есть база знаний по Таро.
- Реализованы основные сущности пользователя.
- Есть Docker.
- Есть задел под монетизацию и админ API.

Минусы:
- Проект выглядит как прототип.
- Многие функции заявлены в README, но реализованы частично либо отсутствуют.
- Есть признаки незавершенного рефакторинга.
- Недостаточное покрытие тестами.
- Отсутствуют миграции БД.
- Не реализована полноценная платежная логика.
- Не реализована production-безопасность.

# Критические проблемы

## 1. В репозитории присутствует .env

Файл:

    astro-bot/.env

Проблема:
- секреты попали в репозиторий;
- риск компрометации токенов и API-ключей.

Требуется:
- удалить .env из репозитория;
- перевыпустить все токены;
- оставить только .env.example.

Приоритет: CRITICAL

---

## 2. Обнаружены pyc-файлы

Присутствуют:

    __pycache__
    *.pyc

Проблема:
- мусор в репозитории;
- увеличение размера;
- нарушение best practices.

Требуется:
- удалить;
- добавить правила в .gitignore.

Приоритет: HIGH

---

## 3. Нет Alembic миграций

Есть каталог:

    migrations/

Но фактически схема БД создается через:

    Base.metadata.create_all()

Проблема:
- невозможно безопасно обновлять продакшен БД.

Требуется:
- внедрить полноценные миграции Alembic;
- создать initial migration.

Приоритет: HIGH

---

## 4. MemoryStorage

Используется:

    MemoryStorage()

Проблема:
- потеря состояний после рестарта;
- не подходит для production.

Требуется:
- RedisStorage.

Приоритет: HIGH

---

## 5. Admin API защищен слабо

Сейчас ключ вычисляется из BOT_TOKEN.

Проблема:
- компрометация токена = полный доступ к API.

Требуется:
- отдельный ADMIN_API_KEY;
- JWT;
- IP whitelist для админки.

Приоритет: CRITICAL

---

# Архитектурные проблемы

## 6. Смешение слоев

В handlers присутствует бизнес-логика.

Пример:

- загрузка карт
- выбор карт
- формирование интерпретаций

Должно быть:

handlers
↓
services
↓
repositories

Требуется вынести всю бизнес-логику в сервисы.

Приоритет: HIGH

---

## 7. Отсутствует Repository Layer

Есть каталог:

    repositories/

Но фактически не используется.

Требуется:

UserRepository
HistoryRepository
PaymentRepository

Приоритет: MEDIUM

---

## 8. Нет Dependency Injection

Много прямых импортов:

    async_session
    settings

Требуется:
- внедрить DI;
- избавиться от глобальных зависимостей.

Приоритет: MEDIUM

---
